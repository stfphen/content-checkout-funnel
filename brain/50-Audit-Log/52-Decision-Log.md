# Decision Log

The "why" behind how things are built. Append new decisions at the top with a date.

| Date | Decision | Rationale |

| 2026-07-04 | **DGTL Group gets a fourth template `agency` (`components/agency/` + `Agency.module.css` on isolated `--ag-*` vars), not a showcase reuse.** | The brief's rule: reuse showcase only if section shapes genuinely fit. They don't — showcase's order is hardcoded music/video sections (VideoWall, MinuteOfMusic, LiveShowcase, ShortFilms) while DGTL needs results-wall / white-label-roster / platform-rail / funding-band. Follows the showcase build pattern exactly (config-driven `agency` block riding sanitize passthrough, sections null when absent, zero funnel-token leak). Locked by `tests/agency-template.test.js`. [[63-Tenants-Catalog]] |
| 2026-07-04 | **Reduced motion in `Reveal`/`Stagger` renders the SAME motion element with zero-duration transitions, never a divergent plain tag.** | The old plain-tag branch caused a hydration mismatch: the server (no media-query knowledge) always renders inline `opacity:0`, the reduced-motion client rendered a different element, React refused to patch, and ALL revealed content (showcase + agency) stayed permanently invisible for `prefers-reduced-motion` users. Identical markup both modes; motion collapses to instant appearance. [[16-Design-System]] |
| 2026-07-04 | **The DGTL Group page is inquiry-led: canonical Content Day package ids/prices, but `action: "booking"` and every card routes into the single Start-a-project form (packageId tagging), no Stripe checkout on this page.** | One CTA label per intent page-wide (taste-skill + goal brief); lead tagging stays consistent with `dgtlmag` because the ids are shared; Stripe stays where the productized funnel already sells. [[63-Tenants-Catalog]] |
| 2026-07-04 | **Generated drafts ship with `fundingPromo.enabled: false` and deterministically cleared rationed eyebrows (warned, not silent).** | End-to-end generation smoke (4 real tenants) showed white-label drafts inheriting the platform default's ENABLED funding cross-promo (another brand's offer + extra section/eyebrow/CTA) and models not reliably obeying the prose eyebrow rule. Hero + packages keep eyebrows; operator can re-enable/re-add anything in the editor. [[16-Design-System]] |
| 2026-07-04 | **Vertical presets have NO default preset; `getVerticalPreset(unknown)` → null, and invalid stored values are deleted (never coerced).** | Absence is the backwards-compat anchor, mirroring premium-agency's empty vars: a tenant without `design.verticalPreset` resolves exactly as before the layer existed (frozen by `tests/section-variants.test.js` snapshots), and there's no fifth "generic" preset to maintain. | 2026-07-04 | **When both a vertical preset and a direction define `sectionOrder`, the preset wins.** | Vertical ordering encodes buyer psychology (trust-first B2B, offer-first trades — see `docs/design-research/`); direction ordering is an aesthetic default. Precedence is one rule everywhere: explicit tenant value > preset > direction > platform default. [[16-Design-System]] |
| 2026-07-04 | **Section variants resolve at render time from preset prefs; the generator emits NO `sectionVariants` (deliberate deviation from master prompt Phase 5.3).** | Stamping variants into config would freeze tenants against preset refinements; render-time resolution restyles every tenant on a preset when prefs improve. Per-tenant pinning stays available via manual patch on `design.sectionVariants`. [[16-Design-System]] |
| 2026-07-04 | **The authority archetype consumes the same `--fp-*` direction tokens (in module-scoped CSS with premium-agency-equivalent fallbacks), not a new token namespace.** | The 15-template matrix is archetype × direction; reusing the closed, tested token set gives authority all 5 looks free with zero `styles.css` regression surface. Showcase deliberately stays on isolated `--sc-*` (verified: zero token leak across directions). [[16-Design-System]] |
| 2026-07-04 | **Funnel eyebrows render only when non-empty; generator rations them (hero + packages only).** | Enables the taste-skill budget (≤1 eyebrow per 3 sections) for library pages. All existing tenants set every eyebrow, so their DOM is unchanged; only the empty-string case (previously an empty `<p class="eyebrow">` gap) differs. [[16-Design-System]] |
| 2026-07-04 | **`.portfolio-card`/`.testimonial`/`.package__badge` got `--fp-*` wraps with their previous literals as fallbacks — an allowed edit to existing selectors.** | These were missed in the original token pass; hardcoded white/rounded values broke dark-cinematic (white card, invisible title) and bold-brutalist (rounded pill on a square page). Fallbacks equal the prior values, so the default path is pixel-identical. [[16-Design-System]] |
| 2026-07-04 | **Professional-services preset does not force a split hero; the paired direction decides.** | Serif display sizes (editorial 76px) cannot fit a 5-word headline in a half-width split column; research pattern B (typographic statement hero + proof banner) fits the vertical equally well. | 2026-07-03 | **Prod tenant seeding is subset-only: `seed-tenants.js --only <slug,…>`; the bare seed is local-dev only.** | Prod rows carry Tenant-Editor edits; a full re-seed silently reverts them (runbook already warned — now enforced by tooling). `--only` validates slugs against the known list and fails loudly on typos. First use: added `dmtv-studio` to prod without touching the other 5 rows. [[41-Deployment-Runbook]] |
| 2026-07-03 | **`$$` in RSC flight payloads is React escaping for `$`-prefixed strings, not data corruption.** | Post-deploy check saw `\"price\":\"$$300+\"` in `__next_f`; source, store, and rendered HTML all show single `$`. React doubles a leading `$` because `$L…`/`$undefined` tokens are references. Do not "fix" stored configs based on payload greps. [[51-Timeline]] |
| 2026-07-03 | **Multiple page templates per tenant via a generic top-level `template` config field + `components/templates/registry.js` (fallback `funnel`), not slug-hardcoded routing or FunnelPage forks.** | Any tenant can adopt any template (no client hardcoding, per product priorities); unknown/unset template resolves to the classic FunnelPage so every existing tenant renders byte-identical.
